## 2. سياق المشروع الحالي

المشروع هو تطبيق Flutter/Dart باسم `firstpro` ووصفه: **الأول برو المحاسبي - تطبيق محاسبي متكامل للأندرويد**.

معلومات مهمة من الفحص الحالي للمشروع:

- المسار المحلي للمشروع: `FirstProAccounting/`.
- المستودع: `https://github.com/Nagmix/FirstProAccounting`.
- الفرع الرئيسي المستهدف: `main`.
- إصدار التطبيق الحالي في `pubspec.yaml`: `2.0.0+2`.
- نطاق Dart SDK: `>=3.3.0 <4.0.0`.
- يعتمد المشروع على Flutter localization مع ملفات `arb` للغة العربية والإنجليزية.
- يستخدم خط Cairo وأصولًا داخل `assets/icons/`.
- قاعدة البيانات تعتمد على SQLite مشفر عبر `sqflite_sqlcipher`.
- توجد بنية بيانات ومهاجرات حتى `migration_v53`.
- يوجد DI عبر `get_it` وإدارة حالة عبر `provider` وبعض ViewModels.
- توجد خدمات محاسبية ومخزون وتقارير مثل:
  - `journal_service.dart`
  - `stock_service.dart`
  - `costing_engine_service.dart`
  - `report_service.dart`
  - `audit_service.dart`
  - `bank_reconciliation_service.dart`
- توجد اختبارات في مجلدات `test/accounting`, `test/database`, `test/regression`, `test/integration`, `test/unit`, `test/widget`.
- GitHub Action الأساسي للبناء والتحقق: `.github/workflows/android-release.yml`.

عند أي تعديل يجب احترام هذه البنية وعدم كسرها.

## 8. مراجعة الأكواد وتصنيف المشاكل

عند مراجعة أي جزء من المشروع يجب تصنيف المشاكل حسب الخطورة:

- 🔴 حرجة `Critical`: تسبب فقدان بيانات، خلل مالي، انهيار التطبيق، ثغرة أمنية خطيرة، أو فشل بناء أساسي.
- 🟠 عالية `High`: تسبب نتائج أعمال خاطئة، مشاكل أداء كبيرة، كسر تدفق رئيسي، أو مخاطرة أمنية واضحة.
- 🟡 متوسطة `Medium`: عيب وظيفي محدود، قابلية صيانة ضعيفة، تكرار مؤثر، أو UX مزعج.
- 🟢 منخفضة `Low`: تحسينات صغيرة، تنظيم، تسمية، أو ملاحظات غير مانعة.

لكل مشكلة اذكر:

- مكانها.
- أثرها.
- السبب المحتمل.
- الحل المقترح.
- أولوية التنفيذ.

### قيود الفحص المحلي الخاصة بهذا المشروع

- لا تقم بتثبيت حزم Flutter محليًا فقط لغرض الفحص.
- لا تعتمد على `flutter pub get` أو تنزيل اعتماديات Flutter محليًا إذا لم تكن متاحة مسبقًا.
- استخدم الفحص المحلي غير المعتمد على تنزيل الحزم عند الإمكان، مثل قراءة الكود، `git diff`, البحث النصي، والتحقق المنطقي.
- التحقق النهائي للبناء والتحليل والاختبارات يكون عبر GitHub Actions بعد الدفع إلى GitHub.
- بعد الدفع، يجب متابعة سجلات GitHub Actions عبر API أو واجهة GitHub وتحليل الأخطاء من السجلات، ثم إصلاحها بدورة جديدة إذا فشل البناء.

## 15. Git و GitHub والدفع للمستودع

عند اكتمال أي مهمة تعديل أكواد أو ملفات مشروع:

- يجب العمل محليًا داخل المشروع أولًا.
- يجب مراجعة الملفات المعدلة قبل commit.
- يجب استثناء الملفات غير المهمة أو التي لا تخص المشروع.
- لا تدفع ملفات build أو cache أو أسرار أو ملفات محلية شخصية.
- يجب أن يكون الدفع إلى الجذر الرئيسي للمستودع على الفرع `main`.
- يجب استخدام اسم المستخدم التالي في Git commits عند الدفع:

```text
Nagmix
```

- يجب استخدام البريد التالي في Git commits:

```text
nooraldeen.ahmed@nagmix.net
```

أوامر الإعداد المحلية المتوقعة قبل commit عند الحاجة:

```bash
git config user.name "Nagmix"
git config user.email "nooraldeen.ahmed@nagmix.net"
```

### التعامل مع بيانات اعتماد GitHub

- لا يتم حفظ GitHub token داخل `AGENTS.md` أو أي ملف في المستودع.
- لا يتم تضمين token داخل remote URL بشكل دائم.
- استخدم بيانات الاعتماد فقط وقت الدفع بطريقة مؤقتة وآمنة.
- إذا لم تتوفر طريقة آمنة لاستخدام بيانات الاعتماد، يجب إيقاف الدفع وطلب طريقة آمنة من المستخدم بدل تخزين السر أو كشفه.
- بعد كل push يجب متابعة GitHub Actions وسجلاتها للتحقق من نجاح البناء والتحليل والاختبارات.
